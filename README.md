# AI text detector (MLOps pipeline)

This is an end to end MLOps pipeline for a classifier that decides whether a passage of text was written by a person or generated by a large language model. You hand it some text, it hands back a label (human or AI), a confidence score, and the version of the model that made the call.

I built it for the MLOps module final pipeline assessment. The pipeline is the implementation of a design I had already sketched earlier in the module on a Padlet, where we were asked to lay out the stages of an ML system. My Padlet had six stages: business problem understanding, data acquisition and management, ML training and testing, model deployment, continuous integration, and continuous monitoring. This repo implements all six and then adds three more: continuous delivery with ArgoCD, continuous training, and explainability.

The wider reason it exists is my MSc dissertation, which is about using AI to help check academic integrity. The dissertation idea is a two stage process. First detect whether a submission looks AI-generated, then generate Bloom's-taxonomy style follow-up questions so a lecturer can check understanding in a short oral. This repo is stage one of that, built as a proper service rather than a notebook.

One more bit of context. When I posted the six-stage design on the Padlet, Shane McDevitt commented that I had left explainability out, and that it really matters when a model might end up affecting someone's grade. He was right, so explainability is now in here as stage nine (the `/explain` endpoint and the attention visualisation behind it).

## What it does

- Base model: `distilbert-base-uncased`, about 66M parameters, fine-tuned for binary classification.
- Data: the HC3 dataset (Human ChatGPT Comparison Corpus) from the HuggingFace Hub, roughly 24k human/ChatGPT answer pairs.
- Input: raw text, up to 512 tokens. Longer text is split into chunks and the scores are averaged.
- Output: `{ label, confidence, model_version }`.
- Tracking: every run logs accuracy, F1 and AUROC plus its hyperparameters to MLflow, and the best model is registered in the MLflow Model Registry under the name `ai-text-detector`.

The one design choice worth calling out, because it is the main thing I changed from the Student-Performance reference repo we were shown in class: the trained model is not baked into the Docker image. The Flask app pulls the current registered model from MLflow when the container starts. That means a new model can go live without rebuilding and repushing the image, which is really the whole point of keeping a registry.

## Rough shape of it

```
 GitHub  ──▶ Actions (CI, train, build/push)
                 │            │
                 ▼            ▼
            MLflow registry   Docker Hub
                 │            │
                 │            ▼
                 │         ArgoCD ──▶ Kind cluster (Flask pods, NodePort 30080)
                 └───────────────────▶ model pulled at pod start
```

Training and MLflow live on a GCP VM. The cluster is a 3-node Kind cluster on the same VM. ArgoCD watches the `k8s/` folder in this repo and syncs any change.

## Layout

```
.github/workflows/   five workflows, one per pipeline stage (01..05)
app/                 Flask serving layer + Dockerfile
training/            HC3 ingestion, preprocessing, DistilBERT training
monitoring/          drift detector that feeds the /metrics endpoint
explainability/      attention-weight visualisation for /explain
tests/               unit, api smoke, and model-regression tests
k8s/                 Kind config + deployment + service
argoCD/              the ArgoCD Application manifest
docs/                architecture, branching, the Padlet stage map, the writeup
```

## Running it locally

```bash
# 1. start MLflow
docker compose up -d mlflow

# 2. pull data and train (logs to MLflow on :5555)
cd training && python ingest.py && python train.py

# 3. build and run the app, pointing it at MLflow
cd ../app && docker build -f Dockerfile -t ai-text-detector:dev ..
docker run -p 5000:5000 -e MLFLOW_TRACKING_URI=http://host.docker.internal:5555 ai-text-detector:dev

# 4. try it
curl -X POST http://localhost:5000/predict -H "Content-Type: application/json" \
     -d '{"text": "Paste a longer passage here so it clears the 20 char minimum."}'
```

You need at least one registered model in MLflow before the app container will come up, because it loads the model on startup.

## Deploying it for real

I run the actual deployment on a GCP VM with a Kind cluster and ArgoCD. The exact commands, pulled from the module tutorials, are in `docs/gcp-setup-checklist.md`. Short version: MLflow runs under docker-compose, training runs on a self-hosted GitHub runner on the VM, images go to Docker Hub tagged by commit SHA, and ArgoCD rolls the cluster when a manifest changes.

## Branching

GitHub Flow, more or less. Short `feature/*` branches off `develop`, `develop` merges up to `main`, and `main` is what ArgoCD deploys. CI runs on every branch. The build-and-deploy workflow only runs on `main`. There is more detail in `docs/branching-strategy.md`.
