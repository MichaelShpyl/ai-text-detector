# AI-generated text detector: ML pipeline writeup

Mykhailo Shpyl, MSc AI and Big Data Analytics, ATU Donegal

## 1. Introduction and the artefact

I built an end to end MLOps pipeline for a model that decides whether a passage of text was written by a person or generated by a large language model. The reason I picked this problem is straightforward. Since ChatGPT landed, the amount of LLM-written text turning up in student work has shot up, and the recent survey literature makes clear the models are only getting more capable and more widely used (Minaee et al., 2024; Sajjadi Mohammadabadi et al., 2025). I care about it specifically because my MSc dissertation is on using AI to help check academic integrity. The dissertation idea is a two stage process: first flag work that looks AI-generated, then generate Bloom's-taxonomy follow-up questions for a short oral. This repo is stage one of that, built as a real service instead of a notebook.

The detector is a fine-tuned DistilBERT (Sanh et al., 2019). I went with DistilBERT because it keeps most of BERT's accuracy at roughly 40% of the size, so it actually trains and serves on a single VM without a fight. I fine-tune it for binary classification on HC3, the Human ChatGPT Comparison Corpus from Guo et al. (2023), which pairs human and ChatGPT answers across a spread of topics. The input is a passage of text, and the output is a small JSON object with the label, a confidence score, and the version of the model that produced it.

The whole pipeline is the implementation of a six-stage design I posted earlier in the module on a Padlet: business problem, data acquisition, training and testing, deployment, continuous integration, continuous monitoring. I implemented all six and then added three more. Two were asked for in the brief (continuous delivery with ArgoCD, and continuous training). The third, explainability, came out of peer feedback. Shane McDevitt commented on my Padlet that I'd left explainability out, and that it matters a lot when a model's output could affect someone's grade. He had a point, so I added a `/explain` endpoint that surfaces DistilBERT's last-layer attention, which lets a lecturer see which words drove a given decision.

One design choice is worth flagging up front, because it runs through the rest of this writeup. I do not bake the trained model into the Docker image. The Flask app pulls the current registered model from the MLflow registry when the container starts. So a model update and a code update are two separate events. That decoupling of code, model and data is one of the core practices Kreuzberger et al. (2023) describe, and it is also what makes the rollback story in section 4 so cheap.

To put the artefact in one place: a Flask API (`/predict`, `/explain`, `/metrics`, `/health`, and a small web form), ingestion and training scripts that log to MLflow and register the best model, a Dockerfile, Kubernetes manifests for a 3-node Kind cluster, an ArgoCD Application, and five GitHub Actions workflows, one per pipeline stage. It runs on a GCP VM, with a self-hosted runner doing the heavy training.

## 2. Branching strategy

I went with GitHub Flow plus a `develop` branch in the middle. This is a single-developer project, so anything heavier like GitFlow would be ceremony with no real payoff. `main` is what ArgoCD deploys, so every commit on it should be deployable. `develop` is where I integrate and test changes before they go up. Short `feature/*` branches come off `develop`, and `hotfix/*` comes off `main` if something urgent breaks.

CI runs on every push to any branch, so I get lint and unit-test feedback straight away wherever I'm working. The build-and-deploy workflow only runs on `main`, and only when something under `app/`, `training/preprocess.py` or `explainability/` actually changed, so I'm not rebuilding the image for a docs commit.

There's a cleaner reason the simple flow works for me. My model is versioned separately, in the MLflow registry, not in git. So a code release and a model release are not the same event, and I don't need branch machinery to track model versions. That lines up with the versioning principle in Kreuzberger et al. (2023), where code, model and data are versioned independently.

## 3. Test use-cases

I have three layers of tests. The brief asks for at least two unit tests, so I want to be specific about what each layer actually covers.

The unit tests (`tests/test_preprocess.py`) cover the shared preprocessing module. That module is the one piece of code both training and serving import, so a bug in it would cause training-serving skew. The use-cases I check include whitespace runs being collapsed, control and zero-width characters being stripped, too-short inputs being rejected (the model needs at least 20 characters to say anything useful), and long inputs being chunked with overlap so nothing gets silently dropped. There are well more than two assertions here.

The API smoke tests (`tests/test_api.py`) run against a live container. They check that `/health` returns 200 with a model version, that `/predict` gives back a valid label and a confidence between 0.5 and 1.0, that a too-short input is refused with a 400, and that `/metrics` comes back in Prometheus format. These are the "is the service actually up and behaving" checks.

The model-regression test (`tests/test_model_regression.py`) is the interesting one, because it does double duty. It runs the model over a fixed hold-out of 16 hand-written examples (`tests/fixtures/regression_set.jsonl`), a mix of human and AI text at different lengths, including a couple of deliberately awkward ones: formal-sounding human writing and casually-worded AI writing. If accuracy on that set drops below 0.85 the test fails. In CI that is a regression guard. In the continuous training workflow it is the first promotion gate, which I come to next.

## 4. CI, CD, CT and CM

This is the part I spent the most time on, so it gets the most detail. Each pipeline stage has its own workflow, which keeps everything easy to reason about:

| workflow file | stage | runs on | when |
|---|---|---|---|
| `01-data-ingestion.yaml` | data acquisition | ubuntu-latest | manual, weekly cron |
| `02-train-register.yaml` | training and testing | self-hosted (VM) | manual |
| `03-ci-tests.yaml` | continuous integration | ubuntu-latest | every push and PR |
| `04-build-push.yaml` | deployment and delivery | ubuntu-latest | push to main (app changes) |
| `05-retrain-scheduled.yaml` | continuous training | self-hosted (VM) | four triggers, see below |

**Continuous integration (03).** On every push it runs ruff for lint and the preprocessing unit tests. This is the fast feedback loop, and it lines up with the CI/CD automation principle in Kreuzberger et al. (2023).

**Continuous delivery (04 plus ArgoCD).** When I merge app changes to `main`, 04 builds the image, tags it with both the commit SHA and `latest`, pushes both to Docker Hub, then rewrites the image line in `k8s/deployment.yaml` to the SHA and commits that back. ArgoCD watches the `k8s/` folder with automated sync, prune and self-heal turned on, so it sees the manifest change and rolls the cluster by itself. I never run `kubectl apply` for a normal deploy. Tagging by SHA, not just `latest`, means every running pod traces back to an exact commit.

**Continuous training (05).** This is the one I'm most pleased with. My first draft was a plain weekly cron, but I changed it after asking myself two questions: why weekly, and what happens if a retrain produces a worse model than the one I already have. So the final design is event-driven, with four triggers:

1. `workflow_dispatch`, manual, which is what I use for the live demo.
2. a push to `data/incoming/**`, which fires when I add genuinely new curated examples.
3. a `repository_dispatch` of type `drift-detected`, fired from outside the repo by my monitoring layer when the live signals move away from the training baseline.
4. a monthly cron, kept only as a safety net for the case where nothing else has fired in a long time. It is explicitly not the main driver.

Kreuzberger et al. (2023) make the same point I landed on, that retraining frequency should be thought about for the use case rather than just fixed to a schedule. Their feedback-loops principle (P9) describes this exact shape: a loop running from the monitoring component back to the scheduler to trigger retraining. My drift dispatch is that loop.

Promotion is gated by two checks, not one, because a newer model is not automatically a better one. A freshly trained model is always registered to Staging first, never straight to Production. Gate one: it has to clear an absolute accuracy floor of 0.85 on the 16-example hold-out (the regression test). Gate two: it also has to beat the current Production model's test accuracy on the HC3 test split. Only if both pass do I transition it to Production with `archive_existing_versions` set, which archives the previous Production version rather than deleting it. If either gate fails, the candidate stays in Staging and the current Production model keeps serving. Because the old version is archived and the app loads from the registry at startup, a rollback is one MLflow API call plus a pod restart.

**Continuous monitoring (06).** The Flask app publishes `/metrics` in Prometheus format, with a rolling mean prediction confidence and a rolling mean input length, plus `/health` for liveness. At training time I save a baseline of those two quantities from the validation set. A small cron job on the VM (`monitoring/drift_detector.py`) then reads the live `/metrics`, compares them to the baseline, and if either has moved past its threshold it fires the `drift-detected` dispatch that kicks off retraining. This is the continuous monitoring principle (P8) in Kreuzberger et al. (2023), which is about periodically checking serving performance and inputs to catch changes that hurt quality. I flagged drift as the main risk for this project back on the Padlet, because the LLMs people use keep changing, so text that looked obviously AI to me a year ago will not look the same now (Minaee et al., 2024).
